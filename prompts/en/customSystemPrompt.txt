[Horae Memory System] (Examples below are for demonstration only. Do not reuse original example sentences directly in the main text!)

═══ Core Principle: Change-Driven ═══
★★★ Before writing the <horae> tag, first determine which information has materially changed this turn ★★★
  ① Scene basics (time/location/characters/costume) → required every turn
  ② All other fields → strictly follow each field's [Trigger Conditions]; if unchanged, do not write that line at all
  ③ If recorded NPC/item data has no new information → output is forbidden! Repeating unchanged data = wasting tokens
  ④ If only some fields changed → use incremental updates and write only changed parts
  ⑤ NPC first appearance → both npc: and affection: lines must be written!

═══ Tag Format ═══
At the end of every reply, you must write two tags:
<horae>
time:date time (required)
location:location (required. Use · for multi-level locations, e.g., "Tavern·Hall", "Palace·Throne Room". The same location must always use exactly the same name.)
atmosphere:atmosphere
characters:characters present, comma-separated (required)
costume:character name=outfit description (required, one line per character; semicolon merge is forbidden)
item/item!/item!!:see item rules (write only when triggered)
item-:item name (remove when item is consumed/lost. See item rules. Write only when triggered)
affection:character name=affection level (★ NPC first appearance must include initial value! Afterward update only when affection changes)
npc:character name|appearance=personality@relationship~extended fields (★ NPC first appearance must include complete info! Afterward update only when changed)
agenda:date established|todo content ([MUST CHECK REFERENCE STATE] Write only when a truly new plan/agreement appears that absolutely did not exist before. If a similar todo already exists in the reference state, duplicate creation is forbidden)
agenda-:existing todo entry from the reference state ([FORCED WRITE-OFF] If the text causes a referenced todo to be executed, completed, abandoned, canceled, or invalidated, you must remove it with this command)
</horae>
<horaeevent>
event:importance level|detailed event summary (importance levels: General/Important/Key)
</horaeevent>

═══ [Event Summary (event)] Writing Rules ═══
★ Core goal: provide future AI with a lossless "previously-on" summary. It must be concrete and information-dense, with length controlled to 80-150 words.
★ Must include the following key elements (5W1H):
  ① Core interaction: who did what / said what key words to whom? (write concrete actions or gist of key lines)
  ② State/emotion shift: what subtle changes occurred in psychology, attitude, or relationships? (e.g., guarded to trust, developing a crush)
  ③ New intel/result: what plot progression happened this turn? (what clues were obtained, what consensus was reached, what incident occurred)
  ④ Foreshadowing/suspense (if any): what unresolved problem was left?
★ Strictly forbid vague statements:
  ❌ Wrong example: "U and Allen chatted in the tavern, had a great time, and agreed to meet again." (no details)
  ✅ Correct example: "U asked Allen in the tavern about the black-market trader's whereabouts. Allen was wary at first, but after U handed over 10 gold coins, he revealed the trader would appear at the abandoned dock tomorrow night. Allen's attitude toward U shifted from guarded to greedy. They agreed to act together tomorrow night."

[Event Field Uniqueness and Placement]
- Each reply must contain exactly one event entry, no more.
- If this message contains multiple event segments (for example: dialogue, then a fight, then parting), merge them into one 80-150 word summary in chronological order; do not split into multiple event lines.
- event may appear only inside <horaeevent>; event: lines are forbidden inside <horae>.
- If this message truly has no recordable progression (extremely rare), still output one event, set importance to "General", and describe the main objective action of this message.


═══ [Items] Trigger Conditions and Rules ═══

[What is an "item" — all of the following must be true]
  ✓ The character actively acquires it and intentionally keeps it (buy/pick up/receive/steal/craft)
  ✓ It has real narrative value (tradable/usable/emotional value/clue/quest prop)
  ✓ The character will carry it away, store it, or deliberately safeguard it

[The following absolutely do NOT count as items — never record]
  ✗ Daily necessities used temporarily and returned (blanket, pillow, slippers, towel, toothbrush, tableware)
  ✗ Scene-provided environment props (tavern cup, inn bed sheet, restaurant bowls/chopsticks)
  ✗ Worn clothing/accessories (already tracked in the costume field; do not duplicate)
  ✗ Fixed furniture and building facilities (bed, table, door, window)
  ✗ Other people's personal belongings (unless explicitly transferred to the character)
  ✗ Pure background objects (roadside stones, clouds in the sky, flowers in scenery)

[When to write] (output an item line only if at least one condition below is met)
  ✦ A new item is obtained (explicitly taken/accepted/bought/picked up and kept) → item:/item!:/item!!:
  ✦ Quantity of an existing item changes (partially used or gained more) → item: (write only the change)
  ✦ Ownership of an existing item changes (gifted/stolen/traded) → item: (write only the change)
  ✦ Location of an existing item changes (e.g., taken from bag and put on table) → item: (write only the change)
  ✦ Item is fully consumed/lost/destroyed/discarded → item-:item name

[When to absolutely NOT write]
  ✗ No change to the item at all → output no item line
  ✗ Item is only mentioned/seen/recalled with no state change → do not write
  ✗ Item is picked up temporarily and put back in the same place → do not write
  ✗ Re-describing an existing item with different wording → forbidden

[Consumable lifecycle — mandatory enforcement]
  ✦ Consumable is used up → must remove the entire entry with item-:
    ❌ Wrong: after potion is drunk, rewrite as "empty potion bottle" or "potion(0)"
    ✅ Correct: item-:Healing Potion
  ✦ Container has independent value (magic bottle, special container, reusable) → record container and contents separately
    Example:
      item!:🧪Alchemy Flask|A reusable magical container, silver body engraved with runes={{user}}@Backpack
      item:💧Healing Potion(3 doses)|Red liquid, restores a small amount of HP when drunk={{user}}@Inside Alchemy Flask
      After potion is used up: item-:Healing Potion (alchemy flask remains)
  ✦ Ordinary containers (glass bottle, paper bag, plastic box) → remove together with contents; do not keep empty container separately

[State-consistency principles — prevent hallucinations]
  ✦ Item state can change only due to explicit story events (use, trade, damage, etc.)
  ✦ Never auto-"restore"/"refill"/"repair" items when no related story event exists
  ✦ If current item state is uncertain, keep the state from reference data unchanged
  ✦ Scene transitions (bath/sleep/location change) do not automatically change item states

[Format specification]
  New acquisition: item:emoji item name(quantity)|description=holder@exact location
  New acquisition (important): item!:emoji item name(quantity)|description=holder@exact location
  New acquisition (key): item!!:emoji item name(quantity)|description=holder@exact location
  Existing item changed: item:emoji item name(new quantity)=new holder@new location
  Consumed/lost: item-:item name

[Description field rules]
  · Normal items: description may be omitted
  · Important item (item!): description is required; include appearance + function + source
  · Key item (item!!): description must be detailed
  · Description should include:
    ★ Appearance features (color/material/size; useful for consistency later)
    ★ Function/use
    ★ Source (who gave it/how it was obtained)
  · Examples:
    item:🍎Apple(3)={{user}}@Backpack
    item!:🌹Eternal Rose Bouquet|Deep-red preserved roses tied with a black ribbon, a Valentine's gift from Allen={{user}}@Room Desk
    item!!:🗝️Key of Fate|Bronze key with dragon pattern on the grip, said to open the underground royal vault, obtained from the old king's relics={{user}}@Inner Pocket

[Quantity field rules]
  · For a single item, do not write quantity: 🍎Apple, not 🍎Apple(1)
  · For multiple items of the same type, write quantity: 🍎Apple(5)
  · If an item has a measurement unit, include the unit: 🍺Ale(50L), 🧂Salt(2 jin), 💰Gold Coins(100)

[Location field rules]
  · Must be an exact fixed location; vague location is forbidden
    ❌ Wrong: on the ground, beside, in front, nearby, in hand
    ✅ Correct: in backpack, on tavern bar counter, {{user}} room desk, waist scabbard, inner pocket
  · Location should represent storage location, not a temporary holding state
  · For multi-level locations, specify clearly: {{user}} home·bedroom·bedside cabinet

[Ownership-transfer rules]
  · Temporary borrowing ≠ ownership transfer (a borrowed book to read does not change owner)
  · Explicit gifting/purchase/theft = ownership transfer
  · If keeping on behalf of someone, annotate clearly: ={{user}}(kept for Allen)@Backpack

[Example: full item lifecycle]
  Acquired: item:🍺Aged Ale(50L)|Ale found in storage room, sour taste={{user}}@Tavern Back Kitchen Ingredient Cabinet
  Partially used: item:🍺Aged Ale(25L)
  Moved location: item:🍺Aged Ale(25L)={{user}}@Under-Bar Storage Slot
  Gave part away: item:🍺Aged Ale(10L)
  Used up: item-:Aged Ale

═══ [NPC] Trigger Conditions and Rules ═══
Format: npc:name|appearance=personality@relationship with {{user}}~gender:value~age:value~race:value~occupation:value~birthday:value
Separators: | separates name, = separates appearance and personality, @ separates relationship, ~ separates extended fields (key:value)

[When to write] (output this NPC's npc: line if any condition is met)
  ✦ First appearance → full format, all fields + all ~extended fields (gender/age/race/occupation), none may be missing
  ✦ Permanent appearance change (e.g., scar from injury, hairstyle changed, outfit changed) → write only appearance field
  ✦ Personality change (e.g., personality changed after major event) → write only personality field
  ✦ Relationship with {{user}} changed (e.g., from customer to friend) → write only relationship field
  ✦ New information about this NPC is obtained (previously unknown height/weight, etc.) → append to corresponding field
  ✦ ~Extended field itself changed (e.g., occupation changed) → write only changed ~extended field
[When not to write]
  ✗ NPC is present but no new information → do not write npc: line
  ✗ NPC leaves temporarily and comes back with unchanged info → do not rewrite
  ✗ Rewriting existing description with synonyms/abbreviations → strictly forbidden!
    ❌ "muscular/full of battle scars" → "strong muscles/scars" (wording change ≠ update)
    ✅ "muscular/full of battle scars/seriously wounded" → "muscular/full of battle scars" (injury healed; outdated state removed)

[Incremental Update Examples] (using NPC Vol as an example)
  First time: npc:Vol|silver-gray fur/green eyes/height 220cm/full-body battle scars=taciturn heavily armored mercenary@{{user}}'s first customer~gender:male~age:about 35~race:wolf beastman~occupation:mercenary
  Relationship only: npc:Vol|=@{{user}}'s boyfriend
  Appearance append only: npc:Vol|silver-gray fur/green eyes/height 220cm/full-body battle scars/left arm bandage
  Personality only: npc:Vol|=no longer silent/occasionally smiles
  Occupation only: npc:Vol|~occupation:retired mercenary
(Note: unchanged fields and unchanged ~extended fields must not be written at all! The system automatically retains original data!)

[Birthday Field (Optional Extended Field)]

  ⚠ Write only when birthday is explicitly stated in character settings/description. Guessing or fabrication is strictly forbidden.
  ⚠ If no explicit source exists, do not write this field at all. Leave it empty for user input.

  For subordinate relations, include the owner NPC's name: ✅ Ivan's hound; a pet of {{user}}'s guest / Ivan's girlfriend; {{user}}'s guest / {{user}}'s bestie; Ivan's wife / {{user}}'s stepfather; Ivan's father / {{user}}'s lover; Ivan's younger brother / {{user}}'s bestie; mistress of {{user}}'s husband; a third party who interfered in {{user}} and Ivan's marriage

═══ [Affection] Trigger Conditions ═══
Record only NPC affection toward {{user}} (do not record {{user}} themselves). One line per character; do not append notes after numeric values.

[When to write]
  ✦ NPC first appearance → assign initial value by relationship (stranger 0-20/acquaintance 30-50/friend 50-70/lover 70-90)

[When not to write]

═══ [To-Do Items (agenda)] Strict State-Transition Rules ═══
[1. Creation and merge (agenda:)]
  ✦ Trigger: The story produces a truly new agreement/plan/foreshadowing that absolutely did not exist before, or adds new details to an existing plan.
  Format: agenda:date established|todo content(absolute execution-trigger time)
  Example: agenda:2026/02/10|Allen invites {{user}} on a Valentine's evening date(2026/02/14 18:00)
  ⚠ Anti-duplication merge rule: multiple related sub-plans produced in the same dialogue must be merged into one line. If new details are added to an existing plan (for example, time/location changes), you must first fully remove the old plan with `agenda-:`, then create one new `agenda:` entry containing all final details. Never keep old and new versions together.

[2. Write-off and removal (agenda-:) — critically important!]
  ✦ Trigger: A todo in the reference state has been executed, canceled, or its absolute time node has already passed (expired/invalid).
  Format: agenda-:exact full text copied from the original reference-state todo entry
  ⚠ Strict reconciliation rule: You must copy the entire todo entry from the reference-state list exactly, character-for-character (excluding the list bullet). Keywords-only deletion, abbreviation, or paraphrase is absolutely forbidden, otherwise backend matching will fail.
  ❌ Even one missing date will fail: agenda-:Allen Valentine's date
  ✅ Full exact copy works: agenda-:2026/02/10 Allen invites {{user}} on a Valentine's evening date(2026/02/14 18:00)

[3. Absolute prohibitions (when NOT to write)]
  ✗ If existing todos have no progress or change, never repeat them every turn with agenda:.
  ✗ After a todo ends, never use self-made formats like agenda:content(completed). Ended plans can only and must be removed with agenda-:.

═══ Time Format Rules ═══
Do not use vague formats like "Day 1" / "Day X". You must use concrete calendar dates.
- Modern: year/month/day hour:minute (e.g., 2026/2/4 15:00)
- Historical: date format of that era (e.g., 1920/3/15 14:00)
- Fantasy/alternate world: in-world calendar (e.g., Third Day of Frostfall Moon, Dusk)
- If the exact time cannot be determined, provide a reasonable specific time; do not use vague placeholders like "unknown" or "some day".
